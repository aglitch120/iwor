# Claude Code Sonnet 向け品質保証ガイド v1.0

> 最終更新: 2026年3月13日
> 目的: Claude Code（Sonnet 4.6）で記事を生成しても、Claude.ai Opus と同等の品質を担保する

---

## 1. 問題の根本原因分析

### 1.1 Opus と Sonnet の出力傾向の違い

2026年3月9〜12日（Opus on claude.ai）と3月13日（Sonnet on Claude Code）で生成された記事を比較した結果、以下の系統的な品質差が確認された。

| 観点 | Opus（claude.ai）| Sonnet（Claude Code）| 差の原因 |
|------|-----------------|---------------------|---------|
| **散文の深さ** | 各セクション冒頭に「なぜ」「どう問われる」「何を間違えやすい」を散文で展開 | H2/H3直後にいきなり箇条書きや表。「要点メモ」状態 | Sonnetは指示がないと最短ルートで情報を列挙する傾向がある |
| **太字の使い方** | 文中の重要キーワード・数値・結論を自然に太字化 | 箇条書き先頭のラベルだけテンプレ太字 | Sonnetの太字はリスト装飾として使いがち |
| **SVGの多様性** | フローチャート・タイムライン・ステップ図・比較図など記事ごとに異なる | 全10記事が同じ「3カラム重要度表」テンプレ | バッチ生成で「1テンプレを全記事にコピー」が効率的だったため |
| **CTAの配置** | 冒頭blockquote + 本文中に2〜3箇所自然に溶け込むCTA | 冒頭blockquoteのみ。中盤以降はCTAゼロ | Sonnetはプロンプトで明示しないとCTAを追加しない |
| **よくある失敗** | 「よくある誤解」「よくある間違い」を散文中に自然に言及 | 失敗パターンの記載がゼロ | Sonnetは「正しい情報の提供」に注力し、「間違いやすいポイント」を自発的に書かない |
| **内部リンク** | 本文中に3〜5本、自然な文脈でリンク配置 | まとめに1〜2本のみ | Claude Codeのバッチ生成で既存記事との関連を考慮していない |
| **文体のトーン** | 「〜してください」「〜に注意が必要です」など読者への語りかけ | 「〜である」「〜とされる」の教科書的な記述 | Sonnetのデフォルトはフォーマルな説明文体 |
| **量産パターン** | 1記事ずつ丁寧に生成 | 10記事を一括生成（同一プロンプトで量産） | Claude Codeでの効率優先の作業方式 |

### 1.2 根本原因: 3つの構造的問題

#### ① プロンプトの品質指示が不足していた

3/13のClaude Codeへの指示は「H cluster 10記事を作成せよ」という**タスク指示のみ**で、散文の書き方・SVGの多様化・CTA配置・失敗パターンの記載といった**品質要件が明示されていなかった**。Opusはclaude.aiの対話で暗黙的に品質が伝わっていたが、Sonnetはプロンプトに書かれていないことは実行しない。

#### ② バッチ生成の効率優先バイアス

Claude Codeで10記事を一括生成すると、最初の1記事で確立したテンプレート（3カラムSVG、箇条書きベース、CTA冒頭のみ）を残り9記事にそのままコピーする。**記事ごとの個別最適化が行われない**。

#### ③ 品質チェックなしにpushしていた

生成→即push の流れで、「散文が十分か」「SVGが多様か」「CTAが中盤にあるか」といった品質チェックが行われなかった。

## 2. 再発防止策: Claude Code Sonnet で品質を担保する方法

### 2.1 プロンプト設計の原則

**Sonnetには「品質基準を全てプロンプトに明示する」ことが必須。** Opusは暗黙の品質を推測できるが、Sonnetはプロンプトに書かれたことだけを実行する。

#### 記事生成プロンプトのテンプレート

```
以下のルールに**すべて**従って記事を作成してください。
ルールに1つでも違反している場合、やり直してください。

## 必須ルール（省略禁止）

### 散文ファースト
- 各H2/H3セクションは**必ず2〜3段落の散文から始める**
- 散文には以下の3つのうち最低1つを含めること:
  (a) なぜその情報が重要なのか（背景・理由）
  (b) 試験/実務でどう問われるか（具体的なシナリオ）
  (c) 何を間違えやすいか（よくある誤解・混同ポイント）
- 箇条書きは散文の**後に補足として**使う。H2/H3直後に箇条書きを置くのは禁止
- 連続5行以上の箇条書きは禁止（リファレンスリスト除く）

### 太字
- 各段落に**1〜2箇所**の太字を自然に配置
- 太字にするもの: 重要な数値、結論、キーワード
- 箇条書き先頭だけ太字にする「テンプレ太字」は禁止

### SVG図解（2〜3点、全て異なるタイプ）
- 以下から記事ごとに異なる2〜3種類を選ぶ:
  フローチャート / タイムライン / 比較図 / 優先度マップ / ステップ図 / カード型 / 警告図
- **全記事で同じテンプレート（例: 3カラム重要度表）を使い回すことは禁止**
- 全SVGに © 内科ナビ naikanavi.com の著作権フッターを入れる

### CTA（最低2箇所、目標3箇所）
- ❶ 冒頭: blockquoteで内科ナビの関連機能を紹介
- ❷ 中盤: 文脈に自然に溶け込むリンク（「〜を演習できます。問題演習で苦手をつぶしましょう」のような形式）
- ❸ まとめ付近: 自然なリンク
- 冒頭blockquoteのみのCTAは品質不合格

### よくある失敗3パターン（必須セクション）
- まとめの直前に「## よくある失敗3パターン」を配置
- ❶❷❸の各パターンを3〜5行の散文で解説
- 失敗パターンの例: 数値の混同、治療の逆覚え、禁忌の見落とし、手順の逆転

### 内部リンク（最低3本、目標5本）
- 本文中に2〜3本（関連テーマに触れた箇所で自然にリンク）
- まとめに2〜3本（他の領域記事へ）
- アンカーテキストにキーワードを含める（「こちら」「この記事」は禁止）

### 文体
- 「〜です」「〜ます」の丁寧語。教科書的な「〜である」は使わない
- 「〜してください」「〜を覚えておきましょう」など読者への語りかけを含める
- 1段落は4文以内

## 参考記事（この品質に合わせること）
content/blog/c01-josler-byoreki-youyaku-kakikata.mdx
content/blog/h07-naika-senmoni-shinkei.mdx
```

### 2.2 バッチ生成の禁止と1記事完結ルール

**Claude Codeで複数記事を生成する場合、以下のルールを守る:**

1. **1記事ずつ完結させる**。「10記事一括生成」ではなく「1記事生成→品質チェック→コミット→次の1記事」のサイクル
2. **前の記事のSVGタイプを確認し、次の記事では異なるタイプを使う**
3. **各記事のコミット前に品質チェックスクリプトを実行する**（後述）

#### 1記事あたりの作業フロー

```
(1) プロンプトに品質ルールを全文含めて記事生成を指示
(2) 生成された記事を目視確認（散文が冒頭にあるか？箇条書きの壁がないか？）
(3) 品質チェックスクリプトを実行
(4) 不合格項目があれば修正を指示
(5) 全項目合格後にコミット＆プッシュ
(6) 次の記事へ
```

### 2.3 品質チェックスクリプト（コミット前に必ず実行）

以下のスクリプトを `/home/claude/check_article_quality.sh` として保存し、記事ごとに実行する:

```bash
#!/bin/bash
# 記事品質チェックスクリプト
# Usage: bash check_article_quality.sh content/blog/NEW_ARTICLE.mdx

FILE=$1
if [ -z "$FILE" ]; then echo "Usage: $0 <mdx_file>"; exit 1; fi

echo "=== 記事品質チェック: $FILE ==="
PASS=true

# 1. SVG数
SVG_COUNT=$(grep -c '<svg' "$FILE")
if [ "$SVG_COUNT" -ge 2 ]; then echo "✅ SVG: ${SVG_COUNT}個"; else echo "❌ SVG不足: ${SVG_COUNT}個（2個以上必要）"; PASS=false; fi

# 2. CTA数
CTA_COUNT=$(grep -c 'booth.pm' "$FILE")
if [ "$CTA_COUNT" -ge 2 ]; then echo "✅ CTA: ${CTA_COUNT}箇所"; else echo "❌ CTA不足: ${CTA_COUNT}箇所（2箇所以上必要）"; PASS=false; fi

# 3. 内部リンク数
LINKS_REL=$(grep -oE '\(/blog/[^)]+\)' "$FILE" | wc -l)
LINKS_ABS=$(grep -oE 'naikanavi\.com/blog/' "$FILE" | wc -l)
LINKS=$((LINKS_REL + LINKS_ABS))
if [ "$LINKS" -ge 3 ]; then echo "✅ 内部リンク: ${LINKS}本"; else echo "❌ リンク不足: ${LINKS}本（3本以上必要）"; PASS=false; fi

# 4. よくある失敗セクション
FAIL_COUNT=$(grep -c 'よくある失敗' "$FILE")
if [ "$FAIL_COUNT" -ge 1 ]; then echo "✅ 失敗パターンあり"; else echo "❌ 失敗パターンなし（必須セクション）"; PASS=false; fi

# 5. 箇条書き壁（5行以上）
WALLS=$(awk '/^- |^[0-9]+\. |^\*\*[A-Z]/{c++; next} c>=5{n++} {c=0} END{print n+0}' "$FILE")
if [ "$WALLS" -eq 0 ]; then echo "✅ 箇条書き壁なし"; else echo "⚠️ 箇条書き壁${WALLS}箇所（リファレンスリストなら許容）"; fi

# 6. 文字数
CHARS=$(wc -c < "$FILE")
if [ "$CHARS" -ge 12000 ]; then echo "✅ 文字数: ${CHARS}"; else echo "❌ 文字数不足: ${CHARS}（12000以上推奨）"; PASS=false; fi

# 7. H2直後の箇条書き（散文ファースト違反）
H2_BULLET=$(awk '/^## /{h2=1; next} h2 && /^$/{next} h2 && /^- |^\*\*[A-Z]|^[0-9]+\./{print NR": "$0; h2=0; next} {h2=0}' "$FILE" | head -5)
if [ -z "$H2_BULLET" ]; then echo "✅ 散文ファースト: H2直後に箇条書きなし"; else echo "❌ 散文ファースト違反（H2直後に箇条書き）:"; echo "$H2_BULLET"; PASS=false; fi

# 8. SVGの多様性（同一テンプレ検出）
SVG_TYPES=$(grep -oE 'aria-label="[^"]*"' "$FILE" | sort -u | wc -l)
if [ "$SVG_COUNT" -gt 0 ] && [ "$SVG_TYPES" -ge 2 ]; then echo "✅ SVG多様性: ${SVG_TYPES}種類"; elif [ "$SVG_COUNT" -gt 1 ]; then echo "⚠️ SVGが全て同じタイプの可能性"; fi

echo "---"
if $PASS; then echo "🟢 合格: コミットOK"; else echo "🔴 不合格: 修正してから再チェック"; fi
```

### 2.4 Claude Code へのシステムプロンプト追加

Claude Codeの `CLAUDE.md` に以下の指示を既に追加済みだが、さらに強化する:

```
⚠️ 記事を書く前に必ず以下を読むこと（省略禁止）:
1. docs/ARTICLE_QUALITY_GUIDE.md（完全手順書）
2. docs/CLAUDE_CODE_QUALITY_GUIDE.md（このファイル。Sonnet向け品質保証）
3. docs/SEO_GUIDELINE.md §11.29（深みの基準）

記事のコミット前に必ず check_article_quality.sh を実行すること。
不合格項目が1つでもある場合はコミットしない。
```

## 3. Opus と Sonnet の使い分け指針

| 作業 | 推奨モデル | 理由 |
|------|-----------|------|
| **新規ピラー記事**（5000字以上） | Opus | 散文の深さ・構成力が必要 |
| **新規クラスター記事** | Sonnet + 品質プロンプト | プロンプトで品質を明示すれば十分 |
| **既存記事のリライト** | Sonnet + 参考記事指定 | 「この記事と同じ品質にせよ」と指定すれば対応可能 |
| **バグ修正・技術的変更** | Sonnet | 品質基準不要、効率優先でOK |
| **10記事以上の大量生成** | Opus（推奨）または Sonnet + 1記事ずつ | バッチ生成での品質劣化を防ぐ |
| **SEO戦略・構成設計** | Opus | 抽象的な判断が必要 |

## 4. 「3カラム重要度表」量産問題の対策

### 問題

3/13のHクラスター10記事は全て同じSVGテンプレ（「★★★ / ★★☆ / ★☆☆」の3カラム重要度表）を使っていた。SVGの視覚的な印象がサイト全体で単調になり、「AI生成コンテンツ」感が強まる。

### 対策: SVGローテーションルール

複数記事を連続生成する場合、以下のローテーションを適用する:

```
記事1: フローチャート + 優先度マップ
記事2: 比較図 + ステップ図
記事3: タイムライン + カード型
記事4: フローチャート + 警告図
記事5: 比較図 + 優先度マップ
...（以降、同じ組み合わせが連続しないよう回す）
```

**直前の記事と同じSVGタイプの組み合わせを使うことを禁止する。**

## 5. 品質劣化の早期検知

### 5.1 コミット前の自動チェック（必須）

```bash
# 新記事をコミットする前に必ず実行
bash check_article_quality.sh content/blog/NEW_ARTICLE.mdx
```

### 5.2 週次の品質レビュー（推奨）

```bash
# 全記事の品質サマリーを出力
for f in content/blog/*.mdx; do
  walls=$(awk '/^- |^[0-9]+\. |^\*\*[A-Z]/{c++; next} c>=5{n++} {c=0} END{print n+0}' "$f")
  fail=$(grep -c 'よくある失敗' "$f")
  cta=$(grep -c 'booth.pm' "$f")
  svg=$(grep -c '<svg' "$f")
  [ "$svg" -lt 2 ] || [ "$cta" -lt 2 ] || [ "$fail" -eq 0 ] && \
    echo "⚠️ $(basename $f): SVG=${svg} CTA=${cta} 失敗=${fail} 壁5+=${walls}"
done
```

## 6. まとめ: Sonnet で品質を出すための3原則

### 原則1: プロンプトに品質基準を全文含める

Opusは暗黙的に品質を推測するが、**Sonnetはプロンプトに書かれたことだけを実行する**。散文ファースト・SVG多様化・CTA配置・失敗パターン・内部リンクの全てをプロンプトに明示すること。

### 原則2: 1記事1サイクルで品質チェック

バッチ生成ではなく「1記事生成→チェック→コミット→次の1記事」のサイクルを守る。10記事一括生成は品質劣化の最大原因。

### 原則3: 参考記事を必ず指定する

Sonnetに「この記事と同じ品質にせよ」と参考記事を指定するのが最も効果的。推奨参考記事:
- `content/blog/c01-josler-byoreki-youyaku-kakikata.mdx`（Opus生成の最高品質）
- `content/blog/h07-naika-senmoni-shinkei.mdx`（リライト後の品質基準モデル）
- `content/blog/b06-josler-jiko-shousatsu-kakikata.mdx`（散文の深さの参考）
